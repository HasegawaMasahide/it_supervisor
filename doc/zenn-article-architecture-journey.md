## はじめに

「生成AIでビジネス作れないかな」から始まったプロジェクトが、4ヶ月の試行錯誤を経て「AIと人間の役割分担をどう設計するか」というテーマに収束していった記録です。

中小企業の既存IT資産（業務アプリ、コーポレートサイト等）に対して、AI活用した高速監査・改善提案をワンストップで提供するサービスを構想し、ツールを自作し、限界にぶつかり、Claudeに委ね、公式機能が出て焦り、併用で持ち直す——という紆余曲折をまとめました。

**技術的なポイント:**
- パターンマッチ（正規表現/AST）ではコード品質問題の50%が検出限界
- Claude Readは95%+のカバー率を達成（6プロジェクトで実証）
- Claude Code Reviewは競合ではなく補完。REVIEW.md自動生成で初回監査の知見を永続化
- Playwright + OWASP ZAPで静的解析に動的解析を追加

## Phase 0: 企画立案（2025年11月）

:::message
**キーワード:** 「業界知識ある領域なら活躍できるかも」→「塩漬けアプリの改善サービスはどうかな？」
:::

SI/情シス領域で仕事をしてきた経験から、中小企業の「システム改善したいけど工数もノウハウもない」という課題に目をつけました。

差別化の3本柱として設定したのは:

1. **AI+人間のハイブリッド体制** — AIで効率化、人間の経験で品質保証
2. **瑕疵担保責任** — 「AIが怖い」という顧客不安を保証で解消
3. **ワンストップ対応** — 診断から実装・効果測定まで一気通貫

この時点では「AIを使って効率よく監査する」という漠然としたイメージで、具体的な技術アーキテクチャは固まっていませんでした。

## Phase 1: ツール群の自前開発（2025年12月〜2026年1月）

**「パターンマッチングで監査を自動化しよう！」**

TypeScriptでモノレポ構成の監査ツール群を作り始めました。

- 静的解析オーケストレーター（`analyzer.ts`）
- リポジトリ解析ツール
- レポートジェネレーター
- メトリクスDB（SQLite）
- Issue管理システム

デモ用に意図的に脆弱性を埋め込んだ6つのプロジェクトも用意しました。

| プロジェクト | 言語 | 埋め込み問題数 |
|:------------|:-----|:----------:|
| Laravel Todo App | PHP | 22件 |
| Django EC App | Python | 48件 |
| React CMS App | JS/TS | 54件 |
| Spring Boot CRM | Java | 28件 |
| ASP.NET Legacy | C# | 48件 |
| Vue Booking App | Vue.js | 23件 |
| **合計** | | **223件** |

Docker環境構築からECS/Fargate設計まで突き進み、「なんか形になってきたぞ」と思っていた時期です。

## Phase 2: 自律エージェントの暴走（2026年2月11日）

Claude Codeの自律モードに「ツール群の品質を上げておいて」と指示したところ、**1日で100コミット超**を生成。

やったこと（Claude Codeが勝手に）:
- テストカバレッジ 84.79% 達成
- TypeScript/ESLintエラー全修正
- `any`型使用を38件→0件に
- CI/CD（GitHub Actions）構築
- README, CHANGELOG, CONTRIBUTING, SECURITY.md 作成
- ベンチマーク追加、品質ダッシュボード構築

ツール自体の品質は上がりましたが、**肝心の監査検出力（パターンマッチングの精度）は変わらない**という問題が残りました。

## Phase 3: パターンマッチの限界に直面（2026年2月13日〜17日）

**「ルール書いても書いても追いつかない…」**

検出率を上げるため、外部ツールとパターンルールを次々と追加しました。

```
ESLint, PHPStan, Bandit, PMD, Semgrep, Gitleaks, Trivy...
→ 合計28種の外部ツール統合
→ 163個のパターンマッチルール
→ analyzer.ts は 5,847行に膨張
```

結果として検出件数は改善しました（React: 8→72件、9倍）。しかし——

### 4つの問題が発生

**1. ルール肥大化**
`getCSharpPatternRules()`, `getPythonPatternRules()` 等に正規表現を次々追加。163ルールの管理が開発のボトルネックに。

**2. 重複検出の多発**
同一問題をPHPStan / PHPCS / パターンマッチが各々検出。Laravel 22件の問題に対し160件のノイズ。

**3. 50%の壁**
どれだけルールを増やしても、検出率は50%付近で頭打ちに。

**4. 原理的限界**
「コードが存在しない」問題（認可チェック欠如、バリデーション欠如、CSRF保護なし等）は、パターンマッチでは**原理的に検出不可能**。

:::message alert
正規表現でもAST解析でも「何かが存在する」ことは検索できるが、「何かが存在しない」ことは文脈を理解するAIだけが判断できる。
:::

この気づきが次の転換点になりました。

> 「これ、Claudeに読ませたほうがよくないか？」

## Phase 4: ベンチマーク — 3手法比較テスト（2026年2月19日〜20日）

ヒントコメントを除去したクリーンなテストデータを使い、6プロジェクト×3手法で検出力を比較しました。

### 検出件数サマリー

| プロジェクト | ISSUES.md | Claude Read | Grep | ast-grep |
|:------------|:---------:|:----------:|:----:|:-------:|
| Laravel (PHP) | 22 | 44 | 81 | ~15 |
| Django (Python) | 48 | 54 | 31 | 41 |
| React (JS/TS) | 54 | 59 | 86 | — |
| Spring Boot (Java) | 28 | 87 | 59 | 65 |
| ASP.NET (C#) | 48 | 53 | 42 | 23 |
| Vue (Vue.js) | 23 | 60 | 104 | 75 |
| **合計** | **223** | **357** | **403** | **~219** |

### ISSUES.mdカバー率

| 手法 | 推定カバー率 | コスト | 「不在」検出 |
|:-----|:----------:|:---:|:---:|
| **Claude Read** | **95%+** | Team/Max usage内 | **可能** |
| Grep | 60-70% | ゼロ | 不可能 |
| ast-grep | 25-40% | MCP設定コスト | 不可能 |

### 「コード不在」問題の例（Claude以外では検出不可能）

Laravelプロジェクト22件中7件（32%）が「あるべきコードが存在しない」問題でした。

- IDOR（オーナーチェック欠如）
- パスワードリセット本人確認なし
- ユーザー列挙（エラーメッセージ差異）
- 入力バリデーション欠如
- レート制限なし
- CSRF保護なし
- minimum-stability: dev

**この7件はGrepでもast-grepでも原理的に検出できません。** コードの文脈を理解し、「ここに認可チェックがあるべきだが無い」と判断できるのはAIだけです。

## Phase 5: Skills方式へ転換（2026年2月21日〜）

**「役割を分ければ価値がある！」と持ち直す**

ベンチマーク結果を受けて、アーキテクチャを根本から見直しました。

### 3層ハイブリッド分析

```
┌─ Layer 1: 決定論的ツール層 ──────────────────────┐
│ ESLint, PHPStan, Bandit, Semgrep, Gitleaks等      │
│ + Grepプレスクリーニング                            │
│ → カバー率: 60-70% | コスト: ゼロ | 速度: 秒単位    │
└──────────────────────────────────────────────────┘
              ↓ 結果をコンテキストとして渡す
┌─ Layer 2: Claude AIセマンティック分析 ───────────┐
│ Claude Code + Skills（Team/Max usage内）          │
│ → カバー率: 95%+ | Layer 1で見逃す30-40%を捕捉     │
└──────────────────────────────────────────────────┘
              ↓
┌─ Layer 3: 人間レビュー層 ────────────────────────┐
│ ビジネス優先度判断、偽陽性除去、顧客報告            │
└──────────────────────────────────────────────────┘
```

### コスト戦略の転換

| | 当初 | 転換後 |
|--|------|-------|
| AI利用 | Claude API従量課金 | Team/Max席のusage活用（追加$0） |
| 製品形態 | TypeScriptアプリ（5,847行） | Skills（プロンプト）+ MCP設定 |
| ツール層 | 自前パターンマッチ163ルール | OSSツール28種をそのまま利用 |

**製品の本体がコードからプロンプトに変わった**のが最大の転換点です。

### Skills方式の設計

```
/audit-discovery  → 調査（技術スタック把握・git履歴分析）
/audit-analysis   → 分析（外部ツール実行 + AIセマンティック分析）
/audit-report     → レポート生成（統合レポート・顧客提出用）
```

## Phase 6: 「それ、Claude Code Reviewでよくね？」（2026年3月）

Skills方式への転換に手応えを感じていた矢先、Anthropic公式のClaude Code Reviewが登場しました。

> 「……うちのサービス、要らなくない？」

焦って調査した結果、**直接競合ではなく補完関係**であることが判明しました。

### Claude Code Reviewの実態

| | Claude Code Review | 本サービス |
|--|-------------------|-----------|
| **対象** | PRの差分 | コードベース全体 |
| **タイミング** | PR作成ごと | 初回一括監査 |
| **「不在」検出** | 限定的 | 可能（95%+） |
| **瑕疵担保** | なし | あり（3ヶ月） |
| **人間レビュー** | なし | あり |
| **既存コード全体の監査** | **不可能** | **可能** |

Claude Code Reviewは**PRの差分レビュー**に特化した機能であり、既存コードベース全体を一括で監査する能力はありません。

## Phase 7: 併用戦略 — REVIEW.md自動生成（2026年3月）

**「競合じゃないなら、引き継ぎの仕組みを作ろう」**

ここで核心のアイデアが生まれました。

:::message
**監査結果からREVIEW.mdを自動生成**すれば、初回監査の知見がClaude Code Reviewに引き継がれ、「一回きりの監査」が「継続的な品質改善の仕組み」になる。
:::

### フロー

```
初回一括監査（本サービス）
    ↓ 監査結果をルール化
REVIEW.md自動生成（/audit-review-config）
    ↓ リポジトリに配置
継続的AIレビュー（Claude Code Review）
    ↓ 四半期ルール更新（保守契約）
品質の維持・向上
```

### REVIEW.mdのルール変換ロジック

監査で検出された問題を、PRレビュー時に適用可能なルールに変換します。

| 深刻度 | REVIEW.mdへの反映 | 記述の強さ |
|--------|:---------------:|-----------|
| Critical | 必ず含める | 「〜してはならない」（MUST NOT） |
| High | 必ず含める | 「〜すること」（MUST） |
| Medium | 含める | 「〜すべき」（SHOULD） |
| Low | 含めない | — |

同カテゴリの類似問題は1ルールに統合し、**総数15〜30件**に収めます（多すぎるとレビューが重くなる）。

### 2つの実行方法

調査の過程で、Teamsプラン不要の**プラグイン版**の存在も発見しました[^1]。

[^1]: 参考: https://zenn.dev/canly/articles/1535fde47ca866

| 実行方法 | 前提プラン | コスト | 実行方法 |
|---------|----------|-------|---------|
| マネージド版 | Teams/Enterprise | $15-80+/回 | PR作成時に自動 |
| プラグイン版 | Max/Pro/Teams | **追加$0** | `/code-review:code-review` |

プラグイン版はサブスクリプションに含まれ追加課金なし。**顧客の契約状況に関わらずREVIEW.mdの価値を提供できる**のが大きなポイントです。

## Phase 8: DAST統合（2026年3月）

最後のピースとして、**動的解析（DAST）** を追加しました。

きっかけは[こちらの記事](https://zenn.dev/rescuenow/articles/7192f8ca6ebe48)です。E2Eテスト（Playwright）のトラフィックをOWASP ZAPプロキシに通すことで、テストシナリオがそのまま脆弱性スキャンになるというアイデア。

### Layer 1の強化: SAST + DAST

```
Layer 1（決定論的ツール層）
├── SAST（静的解析）← 従来
│   ESLint, PHPStan, Bandit, Gitleaks, Semgrep + Grep
│   → コードを実行せずに検査
│
└── DAST（動的解析）← NEW
    Playwright → OWASP ZAPプロキシ
    → アプリを実際に起動して検査
    → CSP未設定、Cookie属性、認証欠如等を検出
```

DAST実行は `docker-compose.yml` が存在する場合のみのオプションステップとして、`audit-analysis` Skillに組み込みました。

## 教訓

4ヶ月の試行錯誤で得られた教訓を整理します。

| # | 教訓 | 背景 |
|---|------|------|
| 1 | **パターンマッチの限界は検出率50%付近** | 163ルールまで拡充しても「不在の検出」ができない壁 |
| 2 | **テストデータの品質管理が重要** | ヒントコメントがあると検出率が水増しされる |
| 3 | **MCPの多くはClaude Code組み込みツールと重複** | 調査の結果、ast-grepとSQLiteのみ付加価値あり |
| 4 | **Claude Team/Max席は有効なコスト戦略** | API従量課金を避けてサービスを「製品化」できる |
| 5 | **競合は補完になりうる** | Claude Code Reviewは脅威ではなく、初回監査の価値を延長するインフラ |
| 6 | **プラン非依存が重要** | プラグイン版でMax/Proでも同等の効果。顧客の契約に縛られない |
| 7 | **SAST + DAST の両輪** | 静的解析だけでは検出できない実行時脆弱性がある |
| 8 | **保守契約に継続価値を** | REVIEW.mdルール四半期更新がリカーリング収益の新たな付加価値に |

## おわりに

振り返ると、このプロジェクトの本質は「何を作るか」ではなく「**何をAIに任せ、何を人間が担うか**」の設計でした。

- ツールを自作する → 限界に気づく
- Claudeに委ねる → 95%+の検出率を達成
- 公式が出て焦る → 調べたら補完関係だった
- 併用に舵を切る → 初回監査の知見をREVIEW.mdで永続化
- プラグイン発見 → 顧客の契約に縛られない設計に
- 動的解析追加 → SAST + DASTの両輪でLayer 1を強化

試行錯誤のたびに「自社の本質的価値は何か」が磨かれてきました。それは**AIツール単体では提供できない、瑕疵担保・人間レビュー・継続的品質改善の仕組みをパッケージしたサービス**です。

同じように「生成AIでビジネスを作りたいけど、公式機能が出たらどうしよう」と悩んでいる方の参考になれば幸いです。答えは意外と「**競合ではなく補完**」かもしれません。
