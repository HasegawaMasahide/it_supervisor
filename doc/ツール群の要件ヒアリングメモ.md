# 再利用可能ツール群 - 要件ヒアリングメモ

## 共通要件

### 全ツール共通で必要なこと
- プロジェクトごとにデータを管理できること
- 複数のプロジェクトを並行して扱えること
- データのエクスポート・インポートができること（別環境への移行用）
- 日本語・英語両方に対応すること（将来の海外展開を見据えて）
- オフライン作業可能（顧客環境に直接持ち込む場合がある）
- ログをしっかり残すこと（瑕疵担保対応時のエビデンス）

### データ保存方針
- SQLiteかPostgreSQLか？ → まずはSQLiteで小さく始める？
- ファイルベースでも良い部分は？（JSON、Markdown等）
- バックアップは自動？手動？

### セキュリティ
- 顧客のソースコードを扱うのでセキュアに
- アクセス制限は必要？（複数人で使う場合）
- 顧客情報の取り扱い（GDPR、個人情報保護法対応）

## Tool 1: メトリクスデータモデル

### やりたいこと
- あらゆるメトリクスを一元管理したい
- 時系列でメトリクスの変化を追跡したい
- Before/After比較を簡単にできるようにしたい
- メトリクスの種類は後から追加できるように（拡張性）

### 収集するメトリクスの例
- セキュリティ：脆弱性件数、CVEレベル別、修正済み/未修正
- パフォーマンス：レスポンスタイム、スループット、リソース使用率
- コード品質：複雑度、重複率、テストカバレッジ
- 技術的負債：未解決Issue数、古いライブラリ数

### 疑問点
- メトリクスのバージョン管理は？（定義が変わったら過去データとの整合性）
- 集計単位は？（ファイル単位、モジュール単位、システム全体）
- 異常値の検出は自動？手動確認？

### データ項目（案）
- プロジェクトID
- 収集日時
- メトリクス種別（カテゴリ）
- メトリクス名
- 値（数値、文字列、真偽値）
- 単位
- 収集元（どのツールで取得したか）
- メモ・備考

## Tool 4: リポジトリ解析ツール

### やりたいこと
- GitリポジトリのURLを渡すだけで自動解析したい
- ディレクトリ構造を把握したい
- 使用言語・フレームワークを自動判定したい
- 依存ライブラリをリストアップしたい
- コミット履歴から開発の活発度を知りたい

### 対応リポジトリ
- Git（GitHub、GitLab、Bitbucket）
- SVN（古い顧客向け）
- ローカルディレクトリ（リポジトリ化されていない場合）

### 解析内容
- ファイル数、総行数
- 言語別の行数・割合
- ディレクトリの深さ
- package.json、composer.json、pom.xml等の解析
- .gitignoreの確認（何が除外されているか）
- READMEの有無

### コミット履歴分析
- 総コミット数
- 最初のコミット日、最後のコミット日
- コミット頻度（月平均）
- 主要なコミッター（誰がメンテしているか）

### 疑問点
- プライベートリポジトリへのアクセスはどう認証する？
- 巨大リポジトリ（1GB以上）の扱いは？
- ブランチは全部見る？mainだけ？

## Tool 5: 静的解析オーケストレーター

### やりたいこと
- 複数の静的解析ツールを一括実行したい
- 言語ごとに適切なツールを自動選択したい
- 解析結果を統一フォーマットで収集したい
- エラーと警告を分けて管理したい

### 実行する解析ツール
**JavaScript/TypeScript**
- ESLint（コード品質）
- Prettier（フォーマット）
- TypeScript Compiler（型チェック）

**PHP**
- PHP_CodeSniffer（コーディング規約）
- PHPStan / Psalm（静的解析）
- PHP Mess Detector（複雑度）

**C#**
- Roslyn Analyzers
- StyleCop

**共通**
- SonarQube（言語横断）
- Snyk（セキュリティ・依存関係）
- Gitleaks（シークレット検出）
- OWASP Dependency-Check（脆弱性）

### 実行方法
- Docker内で実行して環境汚染を防ぎたい
- 並列実行で時間短縮したい
- タイムアウト設定（長時間かかるツールを止める）

### 結果の統一化
- 全ツールの結果を共通フォーマットに変換
- 重複検出の除外（複数ツールが同じ問題を指摘）
- 重要度の正規化（Critical/High/Medium/Low）

### 疑問点
- カスタムルールの追加は？
- false positiveの除外ルール管理は？
- 実行に何時間かかっても良い？（夜間バッチ？）

## Tool 6: メトリクス収集CLI

### やりたいこと
- コマンド一発でメトリクス収集したい
- CI/CDに組み込めるようにしたい
- スクリプトから呼び出せるようにしたい

### コマンド例
```bash
# 基本的な収集
metrics collect --repo ./myapp --project proj-001

# 特定のメトリクスのみ
metrics collect --repo ./myapp --metrics security,performance

# Before/After比較
metrics compare --before ./metrics-before.json --after ./metrics-after.json

# 結果の出力
metrics export --project proj-001 --format json --output ./output.json
```

### 機能
- 進捗表示（どこまで完了したか）
- エラー時のリトライ
- 部分的失敗の扱い（一部ツールが失敗しても続行）
- ログ出力レベル（verbose、normal、quiet）

### 疑問点
- Windows/Mac/Linux全対応必要？
- バイナリ配布？それともnpm/pip経由？
- 設定ファイルはどこに置く？

## Tool 11: 見積もり計算機

### やりたいこと
- Issueから工数を自動見積もりしたい
- 見積もりの根拠を示したい
- 過去実績から精度を上げたい

### 見積もり要素
- 基本工数（Issue重要度ベース）
- 複雑度係数（コード変更範囲、依存関係）
- リスク係数（不確実性）
- バッファ（20-30%）

### 計算方法
```
見積工数 = 基本工数 × 複雑度係数 × リスク係数 × (1 + バッファ率)
```

### 過去実績の活用
- 実績工数を記録
- 見積もりと実績の差分を分析
- 係数を自動調整（機械学習？）

### 出力
- 最小工数、最大工数、期待工数
- 信頼区間
- 見積もりの根拠（何が工数を押し上げているか）

### 疑問点
- 担当者のスキルレベルは考慮する？
- 複数Issueの合計時の調整は？（同時進行できる場合）
- 見積もりの更新頻度は？
