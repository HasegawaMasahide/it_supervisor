# 監査ツール検出率改善計画

## 概要

6つのデモプロジェクトに対する監査ツールの検出率検証の結果、全体検出率は**36.1%**（219件中79件検出）であった。本ドキュメントでは、検出率を目標**70%以上**に引き上げるための具体的な改善計画を優先度別に定義する。

### 現状の検出率

| プロジェクト | 検出率 | 備考 |
|---|:---:|---|
| ASP.NET Legacy System | 64.6% | Roslyn + Gitleaks + OWASP DC が有効に機能 |
| SpringBoot CRM App | 62.5% | Semgrep + OWASP DC のカスタムルールが充実 |
| Vue Booking App | 34.8% | ESLint + パターンルールで一部検出 |
| React CMS App | 27.8% | ESLint 検出あるがカバレッジ不足 |
| Django EC App | 14.6% | ツール有効化済みだが実行時に結果が反映されていない |
| Laravel Todo App | 13.6% | phpcs + 独自ルール3種のみで実行された |

### カテゴリ別検出率

| カテゴリ | 現状平均 | 目標 |
|---|:---:|:---:|
| セキュリティ | 51.6% | 80% |
| 依存関係 | 51.7% | 90% |
| コード品質 | 17.4% | 60% |
| パフォーマンス | 23.9% | 50% |

---

## 改善の全体方針

### 根本原因の分析

検出率が低い主因は「ツールが設定されているが実行時に動作していない」ケースが多い。

1. **ツール実行失敗の未検知**: Gitleaks, Semgrep, PHPMD 等がスクリプト上は有効だが、Docker未起動・バイナリ未インストール等の理由で実行時にサイレントに失敗している
2. **パターンルールのカバレッジ不足**: 88ルール定義済みだが、各デモの問題パターンを網羅できていない
3. **ツール出力パース不備**: ツールが結果を出力していてもIssueに変換されていないケースがある
4. **フレームワーク固有ルールの不足**: Django テンプレート、Laravel Blade の固有パターンが不十分

### 改善アプローチ

```
Phase 1（最優先）: ツール実行の信頼性確保 + Django/Laravel 検出力強化
Phase 2（高優先）: コード品質メトリクス + XSS/eval 検出ルール拡充
Phase 3（中優先）: コードクローン検出 + 依存関係スキャン統一
Phase 4（継続改善）: 検出率の自動検証 + 回帰テスト体制構築
```

## Phase 1: 最優先 — ツール実行信頼性 + Django/Laravel 検出力強化

**目標**: Django 14.6% → 50%、Laravel 13.6% → 50%
**期間**: 2週間
**影響範囲**: `packages/static-analyzer/src/analyzer.ts`、各 `scripts/run-audit-*.ts`

### 1.1 ツール実行失敗の検知・レポート機能

**問題**: ツールがスクリプト上で有効化されていても、実行時にサイレントに失敗し結果が0件になるケースが頻発。Django スクリプトでは Gitleaks, pip-audit, Pylint, Radon がすべて有効化済みだが、出力レポートには Bandit と独自パターンの結果しか反映されていなかった。

**対策**:

#### 1.1.1 ツール実行結果の成否ログ強化

`packages/static-analyzer/src/analyzer.ts` の `runTool()` メソッドに以下を追加:

- 各ツールの実行ステータス（成功/失敗/タイムアウト/スキップ）をレポートに含める
- 失敗時のエラーメッセージ（stderr）をレポートの「ツール実行状況」セクションに出力
- 検出件数が0件のツールに対して警告を出す（ツール自体が動作していない可能性のフラグ）

```typescript
// レポートに含めるべき情報
interface ToolExecutionStatus {
  tool: AnalyzerTool;
  status: 'success' | 'failed' | 'timeout' | 'skipped' | 'not_installed';
  issueCount: number;
  executionTimeMs: number;
  errorMessage?: string;
}
```

#### 1.1.2 ツール存在チェック（pre-flight check）

監査実行前に、有効化されたツールのバイナリ存在を確認する pre-flight check を追加:

```typescript
// 各ツールのコマンド存在確認
const toolCommands: Record<AnalyzerTool, string[]> = {
  [AnalyzerTool.Gitleaks]: ['gitleaks', 'version'],
  [AnalyzerTool.Semgrep]: ['semgrep', '--version'],
  [AnalyzerTool.Bandit]: ['bandit', '--version'],
  [AnalyzerTool.Pylint]: ['pylint', '--version'],
  [AnalyzerTool.PipAudit]: ['pip-audit', '--version'],
  [AnalyzerTool.Radon]: ['radon', '--version'],
  [AnalyzerTool.PHPStan]: ['phpstan', '--version'],
  [AnalyzerTool.PHPMessDetector]: ['phpmd', '--version'],
  [AnalyzerTool.PHPCPD]: ['phpcpd', '--version'],
  [AnalyzerTool.ComposerAudit]: ['composer', 'audit', '--help'],
  // ...
};
```

- バイナリが見つからない場合: Docker fallback を試行し、それも不可ならレポートに「未インストール」と明示
- Docker fallback 成功時: `[Docker]` プレフィックス付きで実行結果を記録

#### 1.1.3 実行結果サマリーの追加

レポートの冒頭に「ツール実行サマリー」セクションを追加:

```markdown
## ツール実行サマリー

| ツール | ステータス | 検出件数 | 実行時間 |
|--------|-----------|---------|---------|
| Bandit | ✅ 成功 | 3件 | 2.1s |
| Pylint | ❌ 失敗（not installed） | 0件 | - |
| Gitleaks | ✅ 成功（Docker） | 5件 | 8.3s |
| pip-audit | ⚠️ タイムアウト | 0件 | 60.0s |
```

**修正対象ファイル**:
- `packages/static-analyzer/src/analyzer.ts` — `runTool()`, `analyze()` メソッド
- `packages/report-generator/` — レポートテンプレートに実行サマリーセクション追加

#### 1.2.3 pip-audit / OWASP DC の結果パース確認

Django EC App で requests (2.25.1), Pillow (8.1.0), urllib3 (1.26.4) の既知CVEが見逃されている。

**確認事項**:
- `runPipAudit()` メソッドの出力パースが正しく動作しているか
- requirements.txt のパスが正しく渡されているか
- pip-audit の実行時に `--requirement` オプションが正しく設定されているか
- OWASP DC が Python パッケージに対して動作しているか

**対策**:
- pip-audit の実行ログを詳細出力し、実行成否を確認
- OWASP DC の Python 対応設定を確認（`--enableExperimental` フラグの必要性）
- fallback として requirements.txt を直接パースし、既知の脆弱バージョンリストと照合するパターンルールを追加

```typescript
// DEP-PY-001 〜 DEP-PY-003: Python 依存関係の既知脆弱バージョン
{
  id: 'DEP-PY-001',
  pattern: /requests\s*[=<>!~]+\s*[12]\.\d+\.\d+/,
  severity: 'high',
  category: 'dependency',
  message: 'requestsの古いバージョンが使用されています。CVE-2023-32681等の脆弱性があります',
  filePattern: /requirements.*\.txt$/,
  contextCheck: (lines: string[], index: number) => {
    const match = lines[index].match(/requests\s*[=<>!~]+\s*(\d+\.\d+)/);
    if (match) {
      const [major, minor] = match[1].split('.').map(Number);
      return major < 2 || (major === 2 && minor < 31);
    }
    return false;
  },
}
```

## Phase 2: 高優先 — コード品質メトリクス + XSS/eval 検出ルール拡充

**目標**: コード品質検出率 17.4% → 50%
**期間**: 2週間（Phase 1完了後）
**影響範囲**: `packages/static-analyzer/src/analyzer.ts`、ESLint設定

### 2.1 コード複雑度メトリクスの統合

**現状の問題**: 全6プロジェクトで「巨大メソッド」「循環的複雑度が高い」「God Class」が検出できていない。Radon（Python）、PHPMD（PHP）、PMD（Java）は設定済みだが、結果がIssueとして反映されていない。

#### 2.1.1 Radon の結果パース改善（Python）

**確認事項**:
- `runRadon()` メソッドが CC（Cyclomatic Complexity）結果を正しくパースしているか
- CC ランク C 以上（複雑度 > 10）を Issue として登録しているか

**対策**:
- Radon の `cc` コマンド出力を確認し、パーサーの修正
- 閾値の設定: CC > 10 を Medium、CC > 20 を High として Issue 登録
- Radon の `raw` コマンドで関数行数チェックも追加（100行超を Medium として登録）

```typescript
// Radon CC 結果のIssue変換例
// 入力: "    F 15:0 complex_function - C (12)"
// 出力: Issue { severity: 'medium', message: '循環的複雑度が12です（閾値: 10）。関数の分割を検討してください' }
```

#### 2.1.2 PHPMD の結果パース改善（PHP）

**確認事項**:
- `runPHPMD()` メソッドの実行コマンドと出力形式
- CyclomaticComplexity, ExcessiveMethodLength, ExcessiveClassLength ルールの有効化状態

**対策**:
- PHPMD のルールセット設定を確認（`cleancode,codesize,controversial,design,naming,unusedcode` が有効か）
- XML 出力形式でのパーサー動作確認
- Docker 環境での PHPMD 実行テスト

#### 2.1.3 PMD の結果パース改善（Java）

**確認事項**:
- `runPMD()` メソッドの実行コマンドとルールセット
- CyclomaticComplexity, GodClass, ExcessiveMethodLength のルール有効化

**対策**:
- PMD のルールセットに `java-design`, `java-codesize` が含まれているか確認
- 結果の XML パーサーが Issue 変換を正しく行っているか検証

### 2.2 XSS/eval 検出ルールの拡充

**現状の問題**: 以下のXSSパターンが見逃されている。

| パターン | 見逃しプロジェクト | 現状 |
|---|---|---|
| Blade `{!! !!}` | Laravel | PHP-SEC-003 定義済みだが実行されていない |
| Django `mark_safe()` | Django | PY-SEC-005 定義済みだが .py のみ対象 |
| Django `{% autoescape off %}` | Django | 未定義 → Phase 1 で追加 |
| Django `\|safe` フィルタ | Django | 未定義 → Phase 1 で追加 |
| Thymeleaf `th:utext` | SpringBoot | JAVA-SEC-002 で検出済み ✅ |

### 4.1 検出率自動検証スクリプトの作成

ISSUES.md と監査結果を自動的に突き合わせ、検出率を算出するスクリプトを作成:

```typescript
// scripts/verify-detection-rate.ts

interface DetectionResult {
  project: string;
  totalIssues: number;
  detectedIssues: number;
  missedIssues: number;
  detectionRate: number;
  byCategory: {
    security: { total: number; detected: number; rate: number };
    codeQuality: { total: number; detected: number; rate: number };
    performance: { total: number; detected: number; rate: number };
    dependency: { total: number; detected: number; rate: number };
  };
  bySeverity: {
    critical: { total: number; detected: number; rate: number };
    high: { total: number; detected: number; rate: number };
    medium: { total: number; detected: number; rate: number };
    low: { total: number; detected: number; rate: number };
  };
}
```

**機能**:
1. ISSUES.md をパースし、意図的問題のリストを取得
2. issues-export.csv をパースし、検出結果のリストを取得
3. 問題と検出結果のマッチング（キーワード・ファイル名・行番号ベース）
4. 検出率の算出とレポート出力
5. 前回実行結果との差分表示（改善/悪化の検知）

### 4.2 CI/CD への統合

GitHub Actions で定期的に検出率を検証:

```yaml
# .github/workflows/detection-rate-check.yml
name: Detection Rate Check
on:
  push:
    paths:
      - 'tools/packages/static-analyzer/**'
      - 'tools/scripts/run-audit-*.ts'
  schedule:
    - cron: '0 0 * * 1'  # 毎週月曜日

jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Setup
        run: cd tools && npm ci && npm run build
      - name: Run all audits
        run: |
          for project in aspnet django laravel react springboot vue; do
            npx tsx scripts/run-audit-${project}.ts
          done
      - name: Verify detection rates
        run: npx tsx scripts/verify-detection-rate.ts
      - name: Check thresholds
        run: |
          # 全体検出率が60%未満なら失敗
          rate=$(cat detection-rate-report.json | jq '.overall.rate')
          if (( $(echo "$rate < 60" | bc -l) )); then
            echo "Detection rate $rate% is below threshold 60%"
            exit 1
          fi
```
