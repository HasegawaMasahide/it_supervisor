# ADR-001: モノレポ構造の採用

**Status**: Accepted
**Date**: 2026-02-11
**Author**: IT Supervisor Tools Development Team

## Context

IT資産監査ツールは、複数の関連するパッケージ(repo-analyzer, static-analyzer, sandbox-builder, report-generator, issue-manager, metrics-model, logger)で構成されています。これらのパッケージは密接に関連しており、以下の課題がありました:

- パッケージ間の依存関係管理が複雑
- バージョン管理とリリースの調整が困難
- コード共有とリファクタリングが難しい
- CI/CDパイプラインの重複

## Decision

npm workspacesを使用したモノレポ構造を採用します。

**構成**:
```
@it-supervisor/tools/
├── packages/
│   ├── logger/              # 基盤パッケージ
│   ├── metrics-model/       # データベース層
│   ├── issue-manager/       # データベース層
│   ├── repo-analyzer/       # 分析層
│   ├── static-analyzer/     # 分析層
│   ├── sandbox-builder/     # 実行環境層
│   └── report-generator/    # 出力層
├── tests/                   # E2E統合テスト
├── examples/                # 使用例
└── package.json             # ワークスペースルート
```

## Consequences

### 良い点

1. **統一された依存関係管理**: すべてのパッケージが同じバージョンの共通依存関係を使用
2. **原子的なコミット**: 複数のパッケージにまたがる変更を1つのコミットで実現
3. **共有開発ツール**: ESLint, TypeScript, Vitest, Huskyなどをワークスペース全体で共有
4. **簡素化されたCI/CD**: 単一のワークフローですべてのパッケージをテスト・ビルド
5. **相互参照の容易さ**: パッケージ間の依存関係を`workspace:*`で管理
6. **コード共有**: 共通のユーティリティ(logger)を簡単に共有可能

### 悪い点

1. **初期セットアップの複雑さ**: モノレポ構造の理解が必要
2. **ビルド時間**: すべてのパッケージを一度にビルドする必要がある
3. **リポジトリサイズ**: すべてのパッケージが1つのリポジトリに含まれる

### リスク

- **循環依存の発生**: 緩和策として、明確な依存関係レイヤーを定義し、CONTRIBUTINGに文書化
- **パッケージの独立性喪失**: 緩和策として、各パッケージは独立してnpmに公開可能

## Alternatives Considered

### 1. マルチリポジトリアプローチ
- **メリット**: 各パッケージが完全に独立
- **デメリット**: バージョン管理の複雑さ、コード共有の困難さ
- **不採用理由**: パッケージ間の密接な関係により、調整コストが高すぎる

### 2. Lerna
- **メリット**: 成熟したモノレポツール
- **デメリット**: npm workspacesで十分な機能が提供される
- **不採用理由**: 追加の依存関係を避け、npmネイティブ機能を優先

### 3. Turborepo
- **メリット**: 高速なビルドキャッシュ
- **デメリット**: 学習曲線、プロジェクト規模に対してオーバーキル
- **不採用理由**: 現時点でのパッケージ数(7個)では不要
