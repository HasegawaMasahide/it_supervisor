# REVIEW.md生成 — 継続的品質改善設定

対象監査ディレクトリ: `$ARGUMENTS`

このSkillは、監査結果（Discovery・Analysis・Report）を入力として、顧客リポジトリに配置する **REVIEW.md**（Claude Code Review設定ファイル）を生成します。

---

## 前提条件

以下のファイルが `$ARGUMENTS` ディレクトリに存在すること:
- `01_discovery.md` — 技術スタック情報
- `02_analysis.md` — 検出問題一覧
- `audit-report.md` — 統合レポート（任意、あればより精度向上）

---

## 生成手順

### Step 1: 入力ファイルの読み込み

1. `01_discovery.md` を読み、**技術スタック**（言語、フレームワーク、バージョン）を特定する
2. `02_analysis.md` を読み、**検出された全問題**とその深刻度・カテゴリを把握する

### Step 2: ルール変換

検出された問題を、PRレビュー時に適用可能なルールに変換する。以下の基準に従うこと:

#### 深刻度によるフィルタリング
| 深刻度 | REVIEW.mdへの反映 | 記述の強さ |
|--------|------------------|-----------|
| Critical | **必ず含める** | 「〜してはならない」「〜は禁止」（MUST NOT） |
| High | **必ず含める** | 「〜すること」（MUST） |
| Medium | **含める** | 「〜すべき」「〜を推奨」（SHOULD） |
| Low | **含めない** | — |

#### 変換のルール
- **1つの問題 → 1つの簡潔なルール**（2文以内）
- 同じカテゴリの類似問題は**1つのルールに統合**する（例: 複数のSQLi問題 → 1つの「パラメータバインディング必須」ルール）
- **「コード不在」問題は特に重要** — 認証・認可・バリデーション等の欠如は明確なルールとして記述する
- ルール総数は **15〜30件** を目安とする（多すぎるとレビューが重くなり、少なすぎるとカバーが不足）

#### 問題カテゴリ → ルールへの変換例

**セキュリティ:**
| 検出問題 | REVIEW.mdルール例 |
|---------|----------------|
| SQLインジェクション | SQLクエリにはORMまたはパラメータバインディングを使用すること。文字列結合によるSQL構築は禁止 |
| XSS脆弱性 | ユーザー入力のHTML出力にはエスケープ処理を必ず適用すること。`v-html`, `dangerouslySetInnerHTML`, `Html.Raw`, `th:utext` の使用にはセキュリティレビューが必要 |
| ハードコード認証情報 | パスワード・APIキー・シークレットをソースコードに直接記述してはならない。環境変数または秘密管理サービスを使用すること |
| CSRF保護欠如 | フォーム送信・状態変更APIにはCSRFトークン検証を実装すること |
| 認証・認可欠如 | 新しいエンドポイント・ページには適切な認証・認可チェックを実装すること。未認証アクセスが許可される場合は明示的にコメントで理由を記載すること |
| 入力バリデーション欠如 | ユーザー入力は必ずサーバーサイドでバリデーションすること |
| IDOR | リソースアクセス時にはオーナーシップ確認（認可チェック）を必ず実装すること |
| レート制限欠如 | 認証エンドポイント（ログイン・パスワードリセット等）にはレート制限を実装すること |
| 機密情報ログ出力 | パスワード・トークン・カード情報等の機密データをログに出力してはならない |

**コード品質:**
| 検出問題 | REVIEW.mdルール例 |
|---------|----------------|
| 空のcatch/except | 例外を握り潰してはならない。最低限のエラーログを記録すること |
| God Class（巨大クラス） | 1ファイル500行を超える場合は責務分割を検討すること |
| マジックナンバー | 意味のある定数名を使用すること |

**パフォーマンス:**
| 検出問題 | REVIEW.mdルール例 |
|---------|----------------|
| N+1クエリ | ループ内でのDBクエリは避け、Eager Loading/JOINを使用すること |
| 大量データ一括取得 | 大量データの取得にはページネーションを実装すること |

**依存関係:**
| 検出問題 | REVIEW.mdルール例 |
|---------|----------------|
| 既知脆弱性のある依存 | 新しい依存関係を追加する際は既知の脆弱性がないことを確認すること |
| EOLフレームワーク | フレームワーク・ライブラリのバージョンアップ時はEOL（サポート終了）を考慮すること |

### Step 3: 技術スタック固有ルールの追加

`01_discovery.md` で特定した技術スタックに応じて、フレームワーク固有のルールを追加する:

#### PHP / Laravel
- `$request->input()` ではなく `$request->validated()` を使用すること
- Eloquentのmass assignmentには `$fillable` または `$guarded` を定義すること
- Bladeテンプレートでは `{!! !!}` ではなく `{{ }}` を使用すること

#### Python / Django
- テンプレートでは `|safe` フィルタの使用を避けること
- `cursor.execute()` にはパラメータバインディングを使用すること（f-string/format禁止）
- `DEBUG = True` を本番設定に含めてはならない

#### JavaScript / TypeScript / React
- `any` 型の使用を避け、適切な型定義を使用すること
- `dangerouslySetInnerHTML` の使用にはセキュリティレビューが必要
- useEffectのクリーンアップ関数でタイマー・サブスクリプションを解除すること

#### Java / Spring Boot
- `@Query(nativeQuery=true)` ではパラメータバインディング(`:param`)を使用すること
- パスワード保存には `PasswordEncoder` を使用すること（平文保存禁止）
- トランザクションが必要な操作には `@Transactional` を付与すること

#### C# / ASP.NET
- コントローラーには適切な `[Authorize]` 属性を付与すること
- フォーム送信には `[ValidateAntiForgeryToken]` を付与すること
- 接続文字列・シークレットは `appsettings.json` ではなく User Secrets または環境変数を使用すること

#### Vue.js
- `v-html` の使用を避け、`v-text` または `{{ }}` を使用すること
- Vuex/Piniaのstateは直接変更せず、mutations/actionsを使用すること

### Step 4: 出力ファイルの生成

以下のフォーマットで `$ARGUMENTS/review-config.md` に出力する:

```markdown
# Code Review Configuration

> このファイルはIT資産監査の結果に基づいて自動生成されました（{監査実施日}）。
> リポジトリルートに `REVIEW.md` として配置してください。
> Claude Code Review がPRレビュー時にこのルールを参照します。
>
> **利用方法（いずれか）:**
> - **マネージド版**（Teams/Enterprise）: REVIEW.mdを配置するだけでPR作成時に自動レビュー
> - **プラグイン版**（Max/Pro/Teams）: Claude Codeで `/code-review:code-review` を実行（追加コスト$0）
>
> ルール総数: {N}件（Critical由来: {n1}件、High由来: {n2}件、Medium由来: {n3}件、スタック固有: {n4}件）

## セキュリティ

{Critical/Highのセキュリティ問題から生成されたルールを箇条書き}

## コード品質

{コード品質問題から生成されたルールを箇条書き}

## パフォーマンス

{パフォーマンス問題から生成されたルールを箇条書き}

## 依存関係

{依存関係問題から生成されたルールを箇条書き}

## {フレームワーク名} 固有のルール

{技術スタック固有ルールを箇条書き}

---

*このREVIEW.mdは [IT資産監査・改善サービス](https://example.com) の監査結果から生成されました。*
*四半期ごとのルール更新を推奨します。更新についてはお問い合わせください。*
```

### Step 5: 確認と調整

生成後、以下を確認する:
- ルール総数が15〜30件の範囲に収まっているか（超過する場合はMedium由来を削減）
- 同じ内容のルールが重複していないか
- ルールが具体的かつ検証可能か（曖昧な表現を避ける）

- X-Frame-Options未設定 → 「レスポンスヘッダーにX-Frame-Options/X-Content-Type-Optionsを含めること」
