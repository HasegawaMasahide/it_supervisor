# IT資産監査 — レポート生成フェーズ

対象リポジトリ: `$ARGUMENTS`

Discovery レポート（`01_discovery.md`）と Analysis レポート（`02_analysis.md`）を読み込み、
顧客提出用の統合レポートを生成してください。

## 入力ファイルの読み込み

```
{リポジトリ名}_audit/
├── 01_discovery.md   ← 調査レポート
├── 02_analysis.md    ← 分析レポート
├── audit-report.md   ← 本Skillで生成（顧客提出用）
└── review-config.md  ← 本Skillで生成（REVIEW.mdとしてリポジトリに配置）
```

上記ファイルが `$ARGUMENTS/../{リポジトリ名}_audit/` に存在するはずです。読み込んでください。

## レポート生成指針

### トーン・文体
- **顧客（非エンジニアの経営層も読む）** に理解できる平易な日本語
- 技術用語には括弧書きで補足説明を添える
- 問題の深刻さを過度に煽らず、事実ベースで記載
- 改善提案は具体的かつ実行可能な内容にする

### 構成原則
- **エグゼクティブサマリー** を冒頭に配置（経営層が最初の1ページで概要を掴める）
- **問題は深刻度順** に並べる（Critical → High → Medium → Low）
- **改善提案には工数見積もりの目安** を添える
- **Before/After** のイメージを具体的に示す

## 4. 継続的品質改善の仕組み

本監査の結果に基づき、**REVIEW.md（Claude Code Review設定ファイル）** を作成しました。

### REVIEW.mdとは

リポジトリルートに配置することで、Claude Code Reviewが今回の監査で発見されたパターンの問題を、今後のPull Requestで自動的に指摘します。監査は「一度きり」ではなく、開発チームの日常的なコードレビューに組み込まれます。

### 導入方法（2つの選択肢）

顧客の契約プランに応じて、いずれかの方法で導入できます:

#### 方法A: マネージド版（Teams/Enterprise プラン向け）

PR作成時にAnthropicインフラ上で自動実行されます。

1. 同封の `review-config.md` をリポジトリルートに `REVIEW.md` としてコピー
2. claude.ai 管理画面でClaude Code Reviewを有効化し、対象リポジトリを選択
3. 以降のPull Requestで自動レビューが実施されます

- 実行コスト: $15〜80+/レビュー（PR規模に応じて変動）
- 組織全体での標準化・強制適用が可能

#### 方法B: プラグイン版（Max/Pro/Teams いずれのプランでも利用可）

Claude Codeのローカルセッションから手動実行します。追加コストはサブスクリプションに含まれます。

1. 同封の `review-config.md` をリポジトリルートに `REVIEW.md` としてコピー
2. Claude Code で `/plugin` → Marketplaces → `code-review` をインストール
3. PRのブランチ上で以下のコマンドを実行:
   - `/code-review:code-review` — ローカル出力（ターミナルに結果表示）
   - `/code-review:code-review --comment` — GitHub PR上にコメントとして投稿

- 実行コスト: 追加$0（既存サブスクリプション内）
- PR作成前のローカル事前チェックにも活用可能

### 含まれるルール

- セキュリティ: {N}件（監査で検出されたCritical/Highの脆弱性パターン）
- コード品質: {N}件（保守性・可読性に関するルール）

- ルール更新は保守契約に含まれます。詳しくはお問い合わせください

## REVIEW.md（継続的レビュー設定）の生成

監査レポート（`audit-report.md`）の生成後、同じディレクトリに **`review-config.md`** を追加出力してください。
これは顧客がリポジトリルートに `REVIEW.md` として配置することで、Claude Code Review が以降のPRで自動レビューを行うための設定ファイルです。

### 生成手順

1. `02_analysis.md` から深刻度が Critical / High / Medium の問題を抽出
2. 各問題を「PRレビュー時に検知可能な形式」の簡潔なルールに変換
   - Critical/High → 「〜すること」「〜してはならない」（MUST / MUST NOT）
   - Medium → 「〜すべき」「〜を推奨」（SHOULD）
   - 同カテゴリの類似問題は1ルールに統合
3. `01_discovery.md` の技術スタック情報に基づき、フレームワーク固有ルールを追加
4. ルール総数は **15〜30件** に収める（多すぎるとレビューが重くなる）
5. `review-config.md` として保存

生成の詳細は `/audit-review-config` Skill を参照してください。
